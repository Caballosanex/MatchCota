# Requirements: MatchCota AWS Production Deployment

**Defined:** 2026-04-08
**Core Value:** A new shelter can register at `matchcota.tech`, instantly get `{slug}.matchcota.tech`, and log in with the password they created, fully over HTTPS.

## v1 Requirements (Final Snapshot)

### Infrastructure

- [x] **INFRA-01**: Operator can provision the full production stack from an empty AWS Academy Lab account using Terraform. _(validated in v1.0)_
- [x] **INFRA-02**: Operator can create Route 53 hosted zone for `matchcota.tech` and obtain nameserver values for DotTech delegation. _(debt carried to next milestone)_
- [x] **INFRA-03**: Visitor can resolve `matchcota.tech` and `*.matchcota.tech` to the EC2 Elastic IP through Route 53 A records. _(validated in v1.0)_
- [x] **INFRA-04**: API client can resolve `api.matchcota.tech` to API Gateway custom domain through Route 53 alias record. _(validated in v1.0)_
- [x] **INFRA-05**: Browser and API clients can complete valid TLS handshakes for `matchcota.tech`, `*.matchcota.tech`, and `api.matchcota.tech`. _(validated in v1.0)_
- [x] **INFRA-06**: Operator can deploy VPC networking with 2 public subnets and 2 private subnets plus required route tables and internet gateway. _(debt carried to next milestone)_
- [x] **INFRA-07**: Lambda can run in private subnet and connect to RDS using security-group-only access control. _(debt carried to next milestone)_
- [x] **INFRA-08**: Operator can provision PostgreSQL 15 on `db.t3.micro` (single-AZ, 20GB, 7-day backups) in private subnet. _(debt carried to next milestone)_
- [x] **INFRA-09**: Backend can store and retrieve upload objects from private S3 bucket with block-public-access enabled. _(debt carried to next milestone)_
- [x] **INFRA-10**: Lambda can access S3 through VPC Gateway endpoint without requiring NAT Gateway. _(debt carried to next milestone)_
- [x] **INFRA-11**: EC2 `t3.micro` instance serves only built React static assets via nginx and does not run backend Python server. _(debt carried to next milestone)_
- [x] **INFRA-12**: API Gateway HTTP API routes production API traffic to Lambda backend through custom domain `api.matchcota.tech`. _(debt carried to next milestone)_

### Backend Runtime

- [x] **BACK-01**: Backend runtime includes `mangum` dependency and deployable Lambda entrypoint. _(debt carried to next milestone)_
- [x] **BACK-02**: Lambda handler file (`backend/app/lambda_handler.py`) exposes `handler = Mangum(app, lifespan="off")` and serves FastAPI routes correctly. _(debt carried to next milestone)_
- [x] **BACK-03**: Backend database configuration uses SQLAlchemy `NullPool` when `ENVIRONMENT=production` to prevent Lambda connection pool exhaustion. _(debt carried to next milestone)_
- [x] **BACK-04**: Production CORS allows `https://matchcota.tech` and wildcard tenant frontend origins while rejecting unrelated origins. _(debt carried to next milestone)_
- [x] **BACK-05**: Tenant registration flow removes runtime Route 53 mutation and relies on wildcard DNS architecture. _(debt carried to next milestone)_

### Frontend Production Surface

- [x] **FRONT-01**: Production frontend build excludes `frontend/src/pages/platform/DemoTest.jsx` and `/demo` route. _(debt carried to next milestone)_
- [x] **FRONT-02**: Production frontend build excludes `frontend/src/pages/public/RegisterAnimal.jsx` and `/register-animal` route. _(debt carried to next milestone)_
- [x] **FRONT-03**: Landing page excludes dev API health check UI, tenant list UI, demo CTA, and device-area diagnostics block. _(debt carried to next milestone)_
- [x] **FRONT-04**: Landing page footer uses `info@matchcota.tech` and year `2026`. _(debt carried to next milestone)_
- [x] **FRONT-05**: Frontend production environment file sets `VITE_API_URL=https://api.matchcota.tech`, `VITE_BASE_DOMAIN=matchcota.tech`, and `VITE_ENVIRONMENT=production`. _(debt carried to next milestone)_

### Tenant Onboarding

- [x] **ONBD-01**: Shelter registrant can set admin password and confirm password during registration with client validation (minimum 8 chars, fields must match). _(debt carried to next milestone)_
- [x] **ONBD-02**: Backend tenant creation schema accepts `admin_password` and validates it as required input. _(debt carried to next milestone)_
- [x] **ONBD-03**: Tenant registration creates tenant and first admin user atomically in one transaction. _(debt carried to next milestone)_
- [x] **ONBD-04**: Newly created admin user is linked to the new tenant, uses tenant email, and stores password as secure hash. _(debt carried to next milestone)_
- [x] **ONBD-05**: Successful registration message confirms active URL in format `{slug}.matchcota.tech` and tells user to log in with their email/password. _(debt carried to next milestone)_
- [x] **ONBD-06**: New shelter can open `https://{slug}.matchcota.tech` immediately after registration and authenticate successfully. _(validated in v1.0)_

### Security and Constraints

- [x] **SECU-01**: Deployment uses existing `LabRole` and `LabInstanceProfile` only, without creating custom IAM roles. _(validated in v1.0)_
- [x] **SECU-02**: Production architecture excludes CloudFront, CloudWatch, SES, NAT Gateway, and Multi-AZ RDS. _(validated in v1.0)_
- [x] **SECU-03**: Monthly infrastructure footprint remains within approximately $50 budget target. _(validated in v1.0)_
- [x] **SECU-04**: Production deployment excludes seed and test fixture data; only real tenant records are used. _(validated in v1.0)_
- [x] **SECU-05**: Tenant-isolated backend queries remain tenant-scoped in production and prevent cross-tenant data access. _(debt carried to next milestone)_

### Verification and Operations

- [x] **VERI-01**: Operator can rebuild environment from lab reset using documented Terraform sequence and resume safely after temporary credential expiry. _(validated in v1.0)_
- [x] **VERI-02**: Operator can run post-deploy validation that confirms DNS + HTTPS readiness for apex, wildcard tenant, and API domains. _(validated in v1.0)_
- [x] **VERI-03**: Project owner can create at least one real test shelter via `matchcota.tech` registration and complete register -> subdomain -> login flow. _(validated in v1.0)_
- [x] **VERI-04**: Deployment docs include actionable runbook steps for infrastructure apply, frontend publish, backend publish, and smoke verification. _(validated in v1.0)_
